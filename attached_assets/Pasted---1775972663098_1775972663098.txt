БЛОК 1 — ПАРСИНГ ЗАГРУЖАЕМЫХ ФАЙЛОВ
═══════════════════════════════════════════════════════

1.1. Извлечение наименования из готового ТЗ (DOCX):
Искать по приоритету:
  а) Паттерн: r"Наименование\S*\s*:\s*(.+)"
     Очистить от: закупка, поставка, приобретение,
     расходных, материалов, в интересах, для нужд
  б) Паттерн: r"(?:закупк[уи]|поставк[уи])\s+(.+?)(?:\s+в интересах|\n)"
  в) Первая строка таблицы характеристик без слов "параметр"/"значение"
  г) Если не найдено — поле подсветить жёлтым, не ставить "Свой товар"

1.2. Приведение наименования к единственному числу:
Словарь замен окончаний:
  -ов/-ев/-ей в конце → убрать (мониторов→монитор, приводов→привод)
  -ы/-и в конце после согласной → убрать (клавиатуры→клавиатура)

1.3. ОКПД2 — если не определён автоматически:
  Показать кнопку "Найти ОКПД2" → поиск по наименованию
  в справочнике okpdtr.ru или через текстовый поиск по базе

1.4. Парсинг таблицы из готового ТЗ:
  Искать таблицу в Приложении/Приложение
  Колонки могут называться: Параметр+Значение, 
  Наименование параметра+Значение параметра, №+Характеристика+Значение
  Поддержать все варианты заголовков
  Пропускать строки где значение пустое или "-"

1.5. Если загружен файл с несколькими товарами (прайс):
  Разбить на отдельные позиции
  Показать список с чекбоксами — пользователь выбирает нужные

БЛОК 2 — ГЕНЕРАЦИЯ ТЗ
═══════════════════════════════════════════════════════

2.1. Запрещённые паттерны в генерируемом тексте:
  - Никаких emoji (U+1F300 и выше)
  - Никаких мета-комментариев: "требуется уточнение",
    "необходимо проверить", "уточнить у заказчика",
    "[заполнить]", "[указать]", "TBD", "N/A"
  - Никаких брендов без "или эквивалент":
    Intel, AMD, Nvidia, Realtek, Samsung, Seagate,
    WD, Kingston, Corsair, Dell, HP, Lenovo, Cisco,
    Mikrotik, Hikvision, Dahua, 3Com
  - Никаких точечных значений без модификатора:
    "512 ГБ" → "не менее 512 ГБ"
    "8 ГБ ОЗУ" → "объём ОЗУ не менее 8 ГБ"
  - Никаких субъективных цветов:
    тёмно-серый→серый, графитовый→серый,
    антрацитовый→серый, шампань→бежевый

2.2. Нормативная база — только актуальная:
  НЕ упоминать: ПП №878, ПП №616, ПП №925 (устарели)
  Использовать: ПП №1875 от 23.12.2024
  Для ПО: ПП №1236, реестр Минцифры
  Для ЕАЭС: ПП №719
  Для УСЛУГ: ПП №1875 и №1236 НЕ применяются

2.3. Структура ТЗ должна содержать:
  - Наименование объекта закупки
  - Цель закупки
  - Требования к товару (функциональные, технические, 
    качественные характеристики)
  - Требования к количеству
  - Требования к упаковке и маркировке (для товаров)
  - Требования к гарантийному сроку
  - Место и срок поставки
  - Приложение с таблицей характеристик (для товаров)

2.4. Таблица характеристик в Приложении:
  Формат: № | Наименование параметра | Значение параметра
  Все числовые значения с модификатором ≥ или "не менее"
  Без брендов и моделей

БЛОК 3 — QA-ПРОВЕРКА
═══════════════════════════════════════════════════════

3.1. Автоматически запускать после каждой генерации.

3.2. Проверки (regex, без LLM):
  ERROR (-15 баллов каждая):
    brand_name — бренд без "или эквивалент"
    exact_value — число+единица без ≥/не менее/от..до
    emoji — любые unicode emoji
    meta_comment — заглушки и мета-фразы
    old_law — упоминание ПП №878/616/925

  WARNING (-5 баллов каждая):
    subjective_color — цвет с оттенком
    no_ktru — нет упоминания КТРУ для товаров
    no_warranty — нет гарантийного срока
    no_country — нет страны происхождения

3.3. Отображение результата перед скачиванием:
  score >= 80: зелёная рамка "Документ соответствует требованиям"
  score < 80: красная рамка, список замечаний с пояснениями
  Всегда: кнопка "Исправить автоматически" + "Скачать как есть"

БЛОК 11 — UX И ИНТЕРФЕЙС
═══════════════════════════════════════════════════════

11.1. Баннер триала (только для plan=trial):
  Прогресс-бар: trial_tz_used / 3
  Зелёный(0)→Жёлтый(2)→Красный(3)
  Текст: "Осталось X из 3 ТЗ · N дней"
  Кнопка "Выбрать тариф →"
  При лимите: кнопка "Сгенерировать" → disabled

11.2. При ошибке 402 — модальное окно:
  Текст причины (trial_expired/trial_limit/monthly_limit)
  Кнопка "Выбрать тариф" → /pricing
  Кнопка "Закрыть"

11.3. Кнопка "История" в навигации для всех авторизованных.

11.4. Страница /payment/success:
  "Тариф [name] активирован ✓"
  Кнопка "Начать работу →"

11.5. Форма загрузки файла:
  Если наименование не распознано — поле подсвечено жёлтым,
  tooltip "Уточните наименование товара"
  Если ОКПД2 не определён — кнопка "Найти ОКПД2"

БЛОК 12 — БЕЗОПАСНОСТЬ И ТЕХНИЧЕСКИЙ ДОЛГ
═══════════════════════════════════════════════════════

12.1. Все admin endpoints — проверять role=admin у пользователя.

12.2. Webhook ЮКасса — проверять IP отправителя:
  Белый список: 185.71.76.0/27, 185.71.77.0/27,
  77.75.153.0/25, 77.75.156.11, 77.75.156.35

12.3. Rate limiting на генерацию:
  Не более 10 запросов в минуту с одного IP.

12.4. Загружаемые файлы:
  Проверять MIME-тип (только .docx, .xlsx, .pdf, .txt)
  Максимальный размер: 10 МБ
  Сканировать на наличие макросов в DOCX

12.5. Удалять файлы из /tmp после генерации.

12.6. Все 500-е ошибки логировать с traceback в таблицу error_log:
  timestamp, user_id, endpoint, error_type, message, traceback

12.7. GET /api/health — возвращать {status: ok, db: ok, version}
  Использовать для мониторинга UptimeRobot.
