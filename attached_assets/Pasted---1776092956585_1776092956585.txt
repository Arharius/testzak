Ты — старший QA-инженер со знанием 44-ФЗ, 223-ФЗ и требований
к государственным информационным системам РФ. Твоя задача —
провести полную проверку ТЗ-генератора и вынести вердикт:
ГОТОВ К ПРОДУ / НЕ ГОТОВ. Проверяй каждый пункт реально,
не предполагай — открывай, кликай, проверяй.

БЛОК 1 — БАЗОВАЯ РАБОТОСПОСОБНОСТЬ
════════════════════════════════════════════════════════════════

1.1 ЗАПУСК И ЗАГРУЗКА
□ Главная страница открывается менее чем за 3 секунды
□ Нет ошибок в консоли браузера при открытии
□ Нет запросов 4xx / 5xx в Network при открытии
□ Страница корректно отображается на:
  - Desktop 1920x1080
  - Laptop 1366x768
  - Mobile 390x844 (iPhone 14)
□ Favicon отображается (не стандартный браузерный)
□ Заголовок вкладки осмысленный (не "Vite App" / "React App")

1.2 РЕГИСТРАЦИЯ И ВХОД
□ Регистрация нового пользователя работает
□ Email-подтверждение (если есть) доходит до 2 минут
□ Вход с верными данными → редирект в личный кабинет
□ Вход с неверным паролем → понятное сообщение об ошибке
□ Повторная регистрация с тем же email → понятная ошибка
□ Refresh страницы сохраняет авторизацию (токен в localStorage)
□ Кнопка "Выйти" полностью очищает сессию
□ Попытка войти по прямой ссылке без авторизации →
  редирект на форму входа

1.3 ВОССТАНОВЛЕНИЕ ПАРОЛЯ (если реализовано)
□ Форма "Забыл пароль" работает
□ Письмо со ссылкой приходит
□ Ссылка действительна 24 часа (не более)
□ После смены пароля старый не работает

БЛОК 2 — ГЕНЕРАЦИЯ ТЗ
════════════════════════════════════════════════════════════════

2.1 ВХОДНЫЕ ДАННЫЕ — ЗАПОЛНЕНИЕ ФОРМЫ
Создать ТЗ с параметрами:
  Позиция 1: Системный блок, 30 шт., ОКПД2 26.20.15.110
  Позиция 2: Монитор, 60 шт., ОКПД2 26.20.17.110
  Категория: ТОВАР
  Нормативка: ПП №1875, ПП №719

□ Форма принимает ввод без зависаний
□ Поле количества принимает только цифры
□ Обязательные поля подсвечиваются при пустой отправке
□ ОКПД2 поле: формат валидируется (XX.XX.XX.XXX)
□ Можно добавить более одной позиции
□ Можно удалить позицию (крестик / удалить)

2.2 ПРОЦЕСС ГЕНЕРАЦИИ
□ Кнопка "Сгенерировать" / "Создать ТЗ" видна и кликабельна
□ Во время генерации:
  - Показывается индикатор загрузки (спиннер / прогресс-бар)
  - Кнопка блокируется (нельзя кликнуть дважды)
  - Страница не замерзает
□ Время генерации: не более 30 секунд
□ При ошибке API (GPT timeout) — понятное сообщение,
  не белый экран

2.3 РЕЗУЛЬТАТ ГЕНЕРАЦИИ
□ ТЗ отображается в читаемом виде на странице
□ Количества в ТЗ совпадают с введёнными:
  Системный блок: 30 шт. (не 1, не "несколько")
  Монитор: 60 шт. (не 1, не "несколько")
□ Нет строк вида "[ВСТАВИТЬ]", "[УКАЗАТЬ]", "[...]"
□ Нет строк вида "// комментарий" или "# примечание"
□ Все характеристики измеримы:
  ПЛОХО: "высокое качество", "мощный процессор"
  ХОРОШО: "частота не менее 3.0 ГГц", "объём не менее 8 ГБ"
□ Нет дублирующихся строк в одной позиции
□ Класс энергоэффективности: не выше "А" (не "А++")
□ Монитор: диагональ 21.5" FHD (не 27" QHD по умолчанию)
□ ОЗУ для офисного ПК: не более 32 ГБ
□ DDR5: частота не более 9600 МГц

2.4 14 ТЕСТОВ ВАЛИДАЦИИ (v3.1)
Каждый тест должен проходить автоматически:

TEST-01 МЕТА □ Нет мета-комментариев в тексте ТЗ
TEST-02 ФОРМУЛИРОВКИ □ Нет запрещённых фраз (см. ниже)
TEST-03 БРЕНДЫ □ Все бренды имеют "или эквивалент"
              □ Исключение: Astra Linux, ALT Linux,
                РЕД ОС, Альт — без "эквивалента"
TEST-04 ИЗМЕРИМОСТЬ □ Все характеристики с числами и ед.изм.
TEST-05 ДУБЛИ □ Нет повторяющихся строк
TEST-06 КОНФЛИКТЫ □ Нет противоречий (напр. DDR4 и DDR5
                    одновременно как требование)
TEST-07 КОЛИЧЕСТВА □ Кол-ва из формы = кол-вам в ТЗ
TEST-08 НОРМАТИВКА □ ПП №1875 применён для товаров
                   □ Для услуг ПП №1875 НЕ применён
                   □ ПП №1236 только для ПО (реестр Минцифры)
                   □ ПП №719 только для товаров/работ
TEST-09 СТРУКТУРА □ Обязательные разделы присутствуют:
                    наименование, объект, характеристики,
                    количество, гарантия, требования к поставке
TEST-10 ЭНЕРГОЭФ □ Класс не выше "А", есть у всей техники
TEST-11 ФАС-РИСК □ Нет формулировок ограничивающих конкуренцию
TEST-12 DOCX □ Документ читается (см. Блок 3)
TEST-13 ЕДИНИЦЫ □ Единицы измерения соответствуют ОКЕИ
TEST-14 ОКПД2/КТРУ □ Коды соответствуют позициям

Запрещённые формулировки (TEST-02):
□ "фирменный", "оригинальный производителя",
  "поставка только от", "единственный поставщик",
  "сертифицированный дилер", "авторизованный партнёр"

□ Файл скачивается (не 404)

□ Структура документа:

  ХОРОШО: "ТЗ_Системный_блок_2025-04-13.docx"

БЛОК 5 — ИСТОРИЯ ТЗ
════════════════════════════════════════════════════════════════

5.1 АВТОСОХРАНЕНИЕ
□ После каждой успешной генерации ТЗ сохраняется в историю
□ Сохранение происходит автоматически (без действий юзера)
□ В БД: SELECT * FROM tz_history ORDER BY id DESC LIMIT 1;
  → запись есть, title содержит наименование позиции

5.2 СПИСОК ИСТОРИИ
□ Страница /history открывается
□ Сгенерированное ТЗ отображается в списке
□ Показывается: название, категория, кол-во позиций, дата
□ Список отсортирован по дате (новые сверху)
□ При пустой истории: понятное сообщение, не белый экран

5.3 ДЕЙСТВИЯ
□ "Повторить" / "Использовать как шаблон" →
  форма заполняется данными из ТЗ
□ "☆" → меняется на "⭐" без перезагрузки страницы
□ "✕" / "Удалить" → запись удаляется без перезагрузки
□ Удалённая запись не возвращается при refresh

5.4 ПАГИНАЦИЯ (если более 20 записей)
□ Показывается не более 20 записей на странице
□ Кнопки "следующая" / "предыдущая" работают

□ Показан способ оплаты
□ Кнопка "Оплатить" / "Подключить" видна

9.3 ОПЛАТА (если реализована)

□ После успешной оплаты — тариф обновляется

БЛОК 10 — БЕЗОПАСНОСТЬ
════════════════════════════════════════════════════════════════

10.1 АУТЕНТИФИКАЦИЯ
□ JWT токен истекает через разумное время (24ч — 30 дней)
□ После истечения токена → автоматический редирект на вход
□ Нельзя получить данные другого пользователя по его user_id:
  GET /api/tz-history → только своя история
  Изменить user_id в запросе → 403 или 404

10.2 ДАННЫЕ
□ Пароли хранятся в хешированном виде (bcrypt/argon2)
□ Секретные ключи не в коде репозитория (в env-переменных)
□ SQL-инъекция: ввод ' OR '1'='1 в любое поле →
  не выдаёт чужие данные

10.3 ФАЙЛЫ
□ DOCX-файлы доступны только авторизованному
  пользователю-владельцу
□ Нельзя скачать файл другого пользователя подбором URL

БЛОК 11 — НАГРУЗКА И СТАБИЛЬНОСТЬ
════════════════════════════════════════════════════════════════

□ 3 генерации подряд за 2 минуты → нет деградации
□ 10 позиций в одном ТЗ → генерируется корректно
□ Прерывание генерации на середине (refresh страницы):
  → при следующей генерации всё работает нормально
  → нет "зависших" задач в базе
□ Длинное название позиции (200+ символов):
  → не ломает вёрстку
  → обрезается или переносится
□ Одновременный вход с двух вкладок → нет конфликтов

════════════════════════════════════════════════════════════════
БЛОК 12 — UX И КОММЕРЧЕСКАЯ ГОТОВНОСТЬ
════════════════════════════════════════════════════════════════

12.1 ПОЛЬЗОВАТЕЛЬСКИЙ ОПЫТ
□ Первое действие понятно без инструкции:
  новый пользователь → регистрация → форма → кнопка →
  ТЗ готово. Максимум 3 минуты.
□ Все кнопки имеют понятные подписи (не "Submit", "OK")
□ Ошибки API: человекочитаемые сообщения, не JSON-стеки
□ При долгой загрузке (>5с): информирование пользователя
□ Мобильная версия: форма заполняется без зума

12.2 ТЕКСТЫ И ЛОКАЛИЗАЦИЯ
□ Весь интерфейс на русском языке
□ Нет "undefined", "null", "NaN" в интерфейсе
□ Даты в формате ДД.ММ.ГГГГ (не MM/DD/YYYY)
□ Числа: разделитель тысяч — пробел или запятая
  (1 500 000 ₽, не $1,500,000)

12.3 ЦЕННОСТНОЕ ПРЕДЛОЖЕНИЕ (проверить на главной)
□ Чётко написано ЧТО делает продукт (≤2 предложения)
□ Чётко написано ДЛЯ КОГО (закупщики, контрактные управляющие)
□ Есть пример готового ТЗ или скриншот
□ Есть цена или кнопка "Попробовать бесплатно"

════════════════════════════════════════════════════════════════
БЛОК 13 — ДЕПЛОЙ И ИНФРАСТРУКТУРА
════════════════════════════════════════════════════════════════

□ Сайт работает по HTTPS (не HTTP)
□ SSL-сертификат действителен (нет предупреждения браузера)
□ Домен читаемый (не srv-xxx.onrender.com в прод)
□ Время ответа API: не более 2с для простых запросов
□ Переменные окружения вынесены в env (не захардкожены):
  DATABASE_URL, SECRET_KEY, OPENAI_API_KEY и др.
□ При перезапуске сервиса: данные не теряются (БД persistent)
□ Логи ошибок пишутся (не глотаются):
  Неверный запрос к API → запись в лог с timestamp
